# supabase/

Supabase project artefacts: migrations, RLS policies, seed data, generated types.

## Structure (to be populated)

- `migrations/` — timestamped SQL migrations (`supabase migration new`)
- `seed.sql` — local dev seed data (never run against prod)
- `config.toml` — Supabase CLI config (generated by `supabase init`)

## Rules

- **RLS on every table.** Default deny. Every migration that creates a table must also create its RLS policies in the same migration.
- **Never write service-role-bypass queries from app code.** Service role stays server-side (see [src/lib/db/CLAUDE.md](../src/lib/db/CLAUDE.md)).
- **Shares first.** Access beyond "owner_id = auth.uid()" must go through RLS helper functions backed by the `shares` table (see [src/lib/shares/CLAUDE.md](../src/lib/shares/CLAUDE.md)). Don't embed ad-hoc access checks in policies.
- **Generated types belong in [src/lib/db/types.ts](../src/lib/db/types.ts).** Regenerate with `pnpm supabase:types` after every migration.

## Convention

- Migration naming: `<timestamp>_<verb>_<subject>.sql` — e.g. `20260419120000_create_profiles.sql`.
- Every CREATE TABLE migration ends with `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` and its policies.
- Prefer Postgres functions for complex access logic; keep policies declarative.
